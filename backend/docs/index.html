Introduction

The ChiroReferral Network API is a referral marketplace connecting patients seeking chiropractic care with practitioners within the network. This document is the complete reference for all data schemas, status enums, and API endpoints.

Base URL

https://api.chiroreferral.com/v1

All endpoints are prefixed with /v1

Interactive Docs

http://localhost:3000/docs

Swagger UI available in development

Core Business Flow

Patient submits referral→ Matching engine runs→ Practitioners see feed→ First claim wins→ Token deducted→ PII unlocked→ Treatment→ Feedback
ℹ️
Patient contact details (name, phone, email, address) are hidden from all practitioners until the referral is claimed. Only the claiming practitioner gains access.

Authentication Flow

The API uses RS256 JWT access tokens (15 min TTL) with HttpOnly cookie refresh tokens (7 day TTL).

Access Token

Include in every authenticated request:

Authorization: Bearer <access_token>

Expires in 900 seconds (15 min)

Refresh Token

Stored automatically as HttpOnly cookie. Exchange it at:

POST /v1/auth/refresh

Rotated on every use. Reuse detection invalidates all tokens.

JWT Payload

FieldTypeDescription
subuuidUser ID
rolestringchiropractor or admin
practitioner_iduuidPractitioner record ID (null for admins)
practitioner_statusstringCurrent practitioner status enum
jtiuuidUnique token ID — used for revocation on logout
expunixExpiry timestamp
⚠️
Practitioners with status other than ACTIVE cannot access referral endpoints. Check practitioner_status in the JWT and prompt the user to complete onboarding.

Pagination

All list endpoints use cursor-based pagination. Cursors are stable under concurrent inserts — no page drift.

Request

GET /v1/referrals/available?limit=20&cursor=eyJpZCI6Ii4uLiJ9

Response Shape

{
  "data": [...],
  "pagination": {
    "cursor": "eyJpZ...",  // null on last page
    "has_next": true,
    "limit": 20
  }
}

Pass the returned cursor as a query parameter in the next request. When has_next is false, you have reached the last page.

Error Handling

All errors return a consistent JSON body:

{
  "code":    "NOT_FOUND",          // machine-readable error code
  "message": "Referral not found", // human-readable message
  "details": { ... }               // optional: validation field errors
}
HTTP StatusCodeWhen
400BAD_REQUESTMissing required header or malformed input
401UNAUTHORIZEDMissing, invalid, or expired JWT
402PAYMENT_REQUIREDInsufficient token balance or no active subscription
403FORBIDDENAuthenticated but not permitted (wrong role or inactive status)
404NOT_FOUNDResource does not exist or not accessible to this user
409CONFLICTDuplicate resource, already claimed, or invalid state transition
422VALIDATION_ERRORRequest body / query param failed schema validation
423LOCKEDReferral claim lock contention — retry after ~1 second
429RATE_LIMITEDToo many requests — see rate limit headers
500INTERNAL_ERRORUnexpected server error
503SERVICE_UNAVAILABLEDatabase or Redis dependency down (readiness probe)

Creates a referral from a patient intake form. Triggers geocoding + matching engine. No account required.

ℹ️
Rate limit: 5 requests per 10 minutes per IP. Supply Idempotency-Key header to prevent duplicate submissions on retry.

POST /v1/auth/refresh Exchange refresh token for new access token ▶

No request body. Reads the HttpOnly refresh_token cookie automatically. Rotates refresh token on use.

// Response 200
{ "access_token": "eyJhbGci...", "expires_in": 900 }
POST /v1/auth/logout Revoke current token 🔐 Auth ▶

Revokes the access token JTI and clears the refresh token cookie. Returns 204.

POST /v1/auth/forgot-password Send password reset email ▶

Always returns 204 (prevents email enumeration). Reset link expires in 1 hour.

{ "email": "jane@example.com" }

POST /v1/referrals/available/:referralId/claim ⚡ Claim a referral — atomic, first-wins ▶
🔴
Required header: Idempotency-Key: <uuid> — supply a fresh UUID per claim attempt. Safe to retry with the same key on network failure.
⚠️

Live Feed — Server-Sent Events 🔐 ACTIVE only

GET /v1/referrals/stream Live referral notifications (text/event-stream) ▶

Opens a persistent connection. The server pushes events as they occur.

// Connection established
event: connected
data: {"status":"connected"}

// New referral in your area
data: {"event":"referral_available","referral_id":"uuid"}

// A referral you could see was just claimed by someone else
data: {"event":"referral_revoked","referral_id":"uuid"}

// In-app notification created
data: {"event":"notification","notification_id":"uuid","type":"NEW_REFERRAL_AVAILABLE","title":"New Referral Available"}

// Heartbeat (every 30s — prevents proxy timeout)
: heartbeat
ℹ️
Clients should reconnect automatically on disconnect (built-in browser EventSource behaviour). The retry: field defaults to 3 seconds.

JavaScript Example

const es = new EventSource('/v1/referrals/stream', {
  headers: { Authorization: `Bearer ${accessToken}` }
});
es.addEventListener('message', (e) => {
  const event = JSON.parse(e.data);
  if (event.event === 'referral_available') {
    // show toast / refresh list
  }
});
